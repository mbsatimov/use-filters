## `f.select`

One choice from a fixed, in-memory list. `valueType` is required — it declares
the value type (`options` are checked against it, so a string union stays a
string union, numeric ids stay numbers):

```ts
status: f.select({
  label: 'Status',
  valueType: 'string',
  options: [
    { label: 'Open', value: 'open' },
    { label: 'Closed', value: 'closed' }
  ]
});
// params.status → 'open' | 'closed' | null
```

The resolved filter also exposes `selectedOption` — the full option object for the
current value — so you can render its label without a lookup.

## `f.multiSelect`

Many choices from the same kind of fixed list. The value is an array:

```ts
labels: f.multiSelect({ label: 'Labels', valueType: 'string', options: labelOptions });
// params.labels → V[] | null
```

`selectedOptions` gives you the full option objects for the current selection.
Array-shaped values are joined in the URL with `,` by default — change the
delimiter with [`arraySeparator`](/docs/api/create-filters) if a value can contain a
comma.

## `f.tags`

A freeform list of strings the user types in — no predefined options, no server
lookup. Use it for "tags contains", pasted ids, arbitrary keywords:

```ts
tags: f.tags({ label: 'Tags', placeholder: 'Add a tag…' });
// params.tags → string[] | null
```

<Callout>
  Fetching the options from a server instead of listing them here? Use [`asyncSelect` /
  `asyncMultiSelect`](/docs/filter-types/async), which search server-side as the user types.
</Callout>

---
title: The filter config
description: A config is plain data built with f.* ŌĆö value types, defaults, and hidden filters.
---

The object you pass to `useFilters` is just **data**. Each entry is built with an
`f.*` helper that records the filter's kind and options ŌĆö nothing is rendered, and
nothing runs until the hook resolves it.

```ts
const configs = {
  search: f.text({ label: 'Search' }),
  status: f.select({ label: 'Status', valueType: 'string', options: statusOptions }),
  min_price: f.number({ label: 'Min price' })
};
```

Because it's plain data, a config can be shared (exported from a module),
[built at runtime](/docs/guides/dynamic-filters) from a backend response, or handed
to a [route loader](/docs/guides/route-loaders) ŌĆö the same object drives the hook
and its loader twin.

## Value types

The builder you pick determines the type of that filter's value in `params`:

```ts
f.text(...)         // string | null
f.number(...)       // number | null
f.boolean(...)      // boolean | null
f.select(...)       // V | null       (V driven by valueType, checked against options)
f.multiSelect(...)  // V[] | null
```

For a `select` or `multiSelect`, `V` is driven by the **required** `valueType`
(`'number' | 'string'`) ŌĆö numeric options give `number | null`, a string union
gives `'open' | 'closed' | null`, and `options` are checked against it (a
mismatched option is a compile error). See [Filter types](/docs/filter-types/overview)
for the full list.

<Callout>
  `valueType` is required ŌĆö rather than inferred from `options` ŌĆö so a route loader calling
  `resolveFilterParams` (which never sees `options`) always parses the same type the hook does. See
  [Async filters](/docs/filter-types/async#value-types).
</Callout>

## Defaults

`defaultValue` sets the value used when the URL has no param for that filter. A
filter sitting at its default counts as **inactive** ŌĆö it's excluded from
`isFiltered` and, with `clearOnDefault` (the default), kept out of the URL entirely:

```ts
status: f.select({
  label: 'Status',
  valueType: 'string',
  options: statusOptions,
  defaultValue: 'open'
});
// No `?status` in the URL ŌåÆ params.status is 'open', and isFiltered ignores it.
```

Because a defaulted filter falls back to its default when unset (and again when
cleared), it can **never** be `null` ŌĆö so its value type drops the `| null`:

```ts
f.select({ label: 'Status', valueType: 'string', options }); //                   'open' | 'closed' | null
f.select({ label: 'Status', valueType: 'string', options, defaultValue: 'open' }); // 'open' | 'closed'  (never null)
```

This holds for every kind, and everywhere `params` is derived ŌĆö the hook and
[`resolveFilterParams`](/docs/api/resolve-filter-params) alike. A defaulted
filter is still clearable in the UI: clearing (or `onChange(null)`) resets it to
its default.

## Hidden filters

`hidden: true` keeps a filter's value in `params` (and `filterMap`) but leaves it
out of the `filters` array. Use it for a value you manage somewhere other than the
main toolbar ŌĆö a search box in the header, or a fixed value you set imperatively:

```ts
const { params, filters, filterMap } = useFilters({
  search: f.text({ label: 'Search', hidden: true }), // in params, not in `filters`
  status: f.select({ label: 'Status', valueType: 'string', options: statusOptions })
});
```

More on placing filters in different parts of your UI in
[Search, sort & placement](/docs/guides/search-sort-and-placement).

## Shared per-filter options

Every kind accepts these, regardless of type:

| Option         | Purpose                                                                                    |
| -------------- | ------------------------------------------------------------------------------------------ |
| `label`        | Required. Human label for the control.                                                     |
| `placeholder`  | Optional; defaults to `label`.                                                             |
| `defaultValue` | Value when the URL param is absent (above).                                                |
| `hidden`       | Keep in `params`, omit from `filters` (above).                                             |
| `commit`       | When a change reaches the URL ŌĆö see [Values & commits](/docs/concepts/values-and-commits). |
| `meta`         | Your own typed UI hints ŌĆö see [UI metadata](/docs/guides/ui-metadata).                     |
| `nuqs`         | Per-filter [nuqs options](https://nuqs.dev), e.g. `{ history: 'push' }`.                   |
